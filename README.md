Benchmark runner for evaluating C# solutions generated by local LLMs through an
OpenAI-compatible API such as LM Studio.

The benchmark includes easy, medium and hard C# tasks, hidden MSTest tests,
Docker-based execution, and JSON/Markdown/HTML reports.

## Quick start

1. Copy the example config:

```bash
cp config.example.yaml config.yaml
```

2. Edit `config.yaml` with your LM Studio endpoint, API model, display model
   label, optional company label and quantization label.

3. Build the .NET evaluation image:

```bash
docker build -f docker/Dockerfile.dotnet8 -t csharp-llm-benchmark-dotnet8 .
```

4. Run the full benchmark:

```bash
python -m benchmark.cli run --config config.yaml
```

Run only the easy benchmark:

```bash
python -m benchmark.cli run --config config.yaml --difficulty easy
```

Run only the medium benchmark:

```bash
python -m benchmark.cli run --config config.yaml --difficulty medium
```

Run only the hard benchmark:

```bash
python -m benchmark.cli run --config config.yaml --difficulty hard
```

Run a single task:

```bash
python -m benchmark.cli run --config config.yaml --difficulty easy --task-id easy-001
```

Resume a run from a task, reusing prior `result.json` files from an existing
results folder:

```bash
python -m benchmark.cli run --config config.yaml --resume hard-010
```

When `--resume-dir` is omitted, the runner prompts you to choose a folder under
the configured `benchmark.output_dir`. For non-interactive use, pass it
explicitly:

```bash
python -m benchmark.cli run --config config.yaml --resume hard-010 --resume-dir results/2026-05-16_090131_qwen3.5-4b-q6_k_Q6_K
```

The runner pipelines generation and evaluation: after a task is generated, its
Docker build/tests run in the background while the next task is generated. By
default one evaluation runs at a time. To allow multiple Docker evaluations at
once:

```bash
python -m benchmark.cli run --config config.yaml --evaluation-workers 2
```

Reports are written under `results/<timestamp>_<modelLabel>_<quantization>/`.

## OpenCode generator

The default generator calls the configured OpenAI-compatible API directly. To
run the same tasks through OpenCode instead, build the OpenCode runner image:

```bash
docker build -f docker/Dockerfile.opencode -t csharp-llm-benchmark-opencode .
```

Then set a pinned OpenCode version in `config.yaml`:

```yaml
benchmark:
  generator: "opencode"

opencode:
  version: "1.1.64"
  # Optional, but useful for OpenAI-compatible/custom providers.
  context_limit: 131072
  output_limit: 8192
  compaction:
    auto: true
    prune: true
    reserved: 10000
```

You can also select it from the CLI:

```bash
python -m benchmark.cli run --config config.yaml --generator opencode --opencode-version 1.1.64 --difficulty easy --task-id easy-001
```

OpenCode is installed inside Docker into `opencode.cache_dir`, grouped by
package and pinned version, and is not read from the host machine. The default
cache is `.cache/opencode`, so future runs with the same OpenCode package and
version reuse the existing install. The agent receives only the task public
files, writes the configured `generated_file`, and the normal hidden-test Docker
evaluation still runs separately with networking disabled. If `llm.base_url`
points to `localhost` or `127.0.0.1`, the OpenCode container uses
`host.docker.internal` unless `opencode.container_base_url` is configured.
When `context_limit` and `output_limit` are set, the generated OpenCode config
passes those model limits to the custom provider so automatic compaction can
track the remaining context window. `opencode.compaction` controls OpenCode's
automatic context compaction behavior.

Timed-out OpenCode containers are removed by default. For debugging a stuck
agent, set `opencode.keep_timed_out_containers: true`; the timed-out container
name is then recorded in the task `result.json` under `opencode.container_name`.

## Aggregate benchmark results

To generate Markdown and filterable HTML ranking tables from all benchmark run
summaries under the results directory:

```bash
python benchmark/aggregate_benchmark_results.py
```

By default, this writes `results/benchmark_results.md` and
`results/benchmark_results.html`. Running the command again recreates those
files and overwrites their previous contents. Both reports include the task
with the highest token usage, how many tokens it consumed, and whether the run
used the direct LLM generator or OpenCode. In the HTML report, the
per-difficulty scores and highest-token task columns are hidden by default and
can be shown from the column options above the table. The final score filter
supports a min/max range with a visual slider. The HTML report also includes
final-score and score-vs-time charts that follow the same filters and table
ordering. Clicking a model name toggles the search filter for that model. The
table highlights best visible values in blue and worst visible values in red
for final score, total time, token usage, max task tokens, and tokens/s.

Use `--results-dir` to read runs from a different directory, or `--output` to
write the Markdown report to a different file. The HTML file defaults to the
same path with a `.html` extension, or can be set explicitly with
`--html-output`:

```bash
python benchmark/aggregate_benchmark_results.py --results-dir results --output results/benchmark_results_latest.md --html-output results/benchmark_results_latest.html
```

## Useful commands

List tasks:

```bash
python -m benchmark.cli list-tasks
python -m benchmark.cli list-tasks --difficulty easy
python -m benchmark.cli list-tasks --difficulty medium
python -m benchmark.cli list-tasks --difficulty hard
```

Validate task metadata:

```bash
python -m benchmark.cli validate
python -m benchmark.cli validate --difficulty easy
python -m benchmark.cli validate --difficulty medium
python -m benchmark.cli validate --difficulty hard
```

Run with an API key from the environment:

```bash
LLM_API_KEY=lm-studio python -m benchmark.cli run --config config.yaml
```

Throttle LLM API requests when your provider has a low requests-per-minute
limit. Omit this field, set it to `null`, or set it to `0` to disable the
throttle:

```yaml
llm:
  requests_per_minute: 3
```

Set the API model, display model label, optional company label and quantization
label in `config.yaml`.
`model` is sent to the API, while `modelLabel` is used in result folder names
and aggregate ranking tables. `company` is shown as a separate aggregate table
column. If `modelLabel` is omitted, the model name is used:

```yaml
llm:
  model: "qwen3.5-4b@q4_k_m"
  modelLabel: "qwen3.5-4b"
  company: "Alibaba"
  quantization: "Q4_K_M"
```

You can also set a single task in `config.yaml`:

```yaml
benchmark:
  # Optional: "easy", "medium", "hard" or "all".
  difficulty: "all"
  task_id: "easy-001"
```

## Notes

- The LLM receives a fixed system prompt plus `prompt.md` for each task.
- Hidden tests are copied into an isolated temporary workspace only during
  evaluation.
- The system prompt provides shared output and structure rules: generated C#
  must be a single fenced `csharp` block, avoid C# namespaces, target the
  provided project compiled with the .NET 8 SDK, prefer conservative broadly
  supported C#/BCL APIs, and must not add extra third-party/NuGet packages.
- Medium tasks include data structures, algorithms, async/concurrency, in-memory
  ASP.NET Core APIs, OOP and SOLID exercises.
- Hard tasks add denser deterministic exercises covering interval indexes, LFU
  caching with single-flight loading, dependency planning, async ETL, priority
  scheduling, order matching, in-memory APIs, SQLite inventory, transactional
  persistence, pricing plugins, immutable tries, specification rules and client
  resilience policies.
- API tasks use in-memory HTTP integration tests with
  `Microsoft.AspNetCore.Mvc.Testing`, and the SQLite inventory task uses
  `Microsoft.Data.Sqlite`. Both packages are restored into the Docker image so
  evaluation can continue with container networking disabled.
- Docker is run with no network, read-only root filesystem, dropped
  capabilities, process limits, memory limits, and CPU limits.
